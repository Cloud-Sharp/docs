    파일 업로드 다운로드 전략 체크리스트

    1. 다운로드 명세의 Range 요청 충돌 해소¶

    • "병렬 단일 Range 요청 여러 개 허용"을 명시적으로 기술
    • "한 HTTP 요청 안의 multi-range (Range: bytes=0-99, 200-299)는 미지원"을 명시적으로 기술
    • 두 표현이 문서 내에서 서로 모순 없이 통일되었는지 전수 확인
    • 백엔드·프록시·클라이언트 각 계층의 해석이 동일해지는지 검증

    2. finalize 중복 방지 락 전략 확정¶

    • 락 방식 결정 — DB row lock / Redis 분산 락 / 상태 전이 CAS 중 택일(또는 조합)
    • 선택한 방식의 획득-해제 흐름을 시퀀스로 기술
    • 멀티 인스턴스(수평 확장) 환경에서 중복 finalize 불가능함을 보장하는 근거 명시
    • tus hook 완료 감지 / 별도 완료 엔드포인트 / 백그라운드 finalize 각 경로별로 락 적용 지점 표기
    • idempotency key 또는 상태 전이 조건(예: UPLOADING → FINALIZING CAS) 규칙 문서화

    3. 정리 배치·만료 처리 운영 규칙 수립¶

    • EXPIRED 전환 배치 실행 주기 명시 (예: 매 5분)
    • orphan file 스캔 대상 범위 정의 (임시 디렉터리, 특정 prefix 등)
    • orphan file 판정 기준 정의 (세션 없는 파일, N시간 초과 등)
    • 정리 실패 시 재시도 횟수·백오프 정책 명시
    • 정리 지연·실패에 대한 장애 알림 기준 명시
    • 예약(FileReservation) 해제 조건·타이밍 명시

    4. 업로드 API 계약 보완¶

    • 업로드 상태 조회 API 응답 필드 목록 확정 (상태, 진행률, 에러 등)
    • 사용자 취소 API 규칙 정의 (HTTP 메서드, 상태 전이, 부수 효과)
    • finalize 결과 전달 방식 확정 (동기 응답 / 폴링 / 웹소켓 / 콜백)
    • 에러 HTTP 상태 코드 총괄표 작성 (다운로드 문서 수준으로)
    • tus 전송 계층 에러와 비즈니스 finalize 에러의 구분 기준·응답 형식 정의

    5. quota 판정 공식 확정¶

    • 업로드 시작 허용 조건 공식 확정 (예: used + reserved + expected ≤ allowed)
    • allowed = null(무제한) Space 처리 규칙 명시
    • 동일 Space 다중 탭/세션 동시 업로드 시 예약 경쟁 방지 방법 기술
    • finalize 직전 quota 재검사 실패 시 사용자 에러 메시지·코드 정의
    • 예약 → 확정 전환 시 reserved 감소 · used 증가 원자성 보장 방법 명시

    6. 파일명 충돌 최종 정책 결정¶

    • 사전 검사(FileItem + 활성 FileReservation) 통과 후에도 충돌이 발생하는 시나리오 나열
    • 최종 충돌 처리 정책 택일 — 실패 반환 / 자동 리네임 (1) / 버전 업로드 / 덮어쓰기 모드
    • 선택한 정책의 UX 시나리오 기술 (사용자에게 보이는 메시지·동작)
    • 정책이 API 응답에 어떻게 반영되는지 명시 (새 파일명 반환 등)

    7. 다운로드 세션 권한 변경·강제 revoke 정책¶

    • 세션 발급 후 권한 박탈/공유 링크 폐기 시 즉시 revoke 지원 여부 결정
    • 즉시 revoke 지원 시: revoke 메커니즘 기술 (블랙리스트, DB 상태 체크 등)
    • 즉시 revoke 미지원 시: 짧은 TTL만으로 충분한 근거 및 최대 허용 노출 시간 명시
    • 보안 사고 대응 절차에 세션 강제 만료 단계 포함 여부 확인

    8. 후처리 결과 ↔ 파일 접근 정책 연결¶

    • 바이러스 스캔 완료 전 파일의 다운로드 허용/차단 여부 결정
    • 스캔 실패·감염 탐지 시 FileItem 상태값 정의 (예: quarantined)
    • 썸네일·미리보기 생성 실패 시 사용자 경험 정책 정의 (플레이스홀더, 재시도 등)
    • 후처리 각 단계가 세팅하는 상태값 목록과 전이 시점을 업로드 문서에 명시
    • 다운로드 문서의 차단 조건(삭제/손상/비활성/격리)과 업로드 후처리 상태값이 1:1 매핑되는지 확인

    9. 운영 지표·알림 기준 정의¶

    • 핵심 지표 목록 확정
    • finalize 실패율
    • orphan file 발생 수
    • 평균 finalize 지연 시간
    • 세션 만료율 (업로드 / 다운로드 각각)
    • 다운로드 416 (Range Not Satisfiable) 비율
    • share token / session token 위변조 시도 횟수
    • 각 지표의 정상 범위(SLO) 기준값 설정
    • 경보 임계치 및 알림 채널 정의
    • 대시보드 구성 항목 명시

    10. 스토리지 백엔드별 차이 전략¶

    • Local FS / S3(또는 호환 Object Storage) 각각에 대한 동작 차이 매트릭스 작성
    • 파일 이동: rename 가능 조건 vs copy+delete 폴백 기준 명시
    • copy 중 실패 시 복구·롤백 기준 정의
    • Object Storage의 eventual consistency 고려 여부 및 대응 방법 기술
    • multipart upload 미완료 파트 정리 정책 명시
    • checksum 저장 위치·검증 시점 정의 (DB 컬럼, 메타데이터 헤더 등)
    • storage_key 네이밍 규칙 확정 (prefix, 해시, UUID 등)

    사용법: 각 항목을 전략 문서 개정 시 하나씩 체크하며 닫아 나가면, 리뷰에서 지적된 10개 영역이 빠짐없이 반영됩니다.
