    다운로드 세션 권한 변경 및 Revoke 정책¶

    문서 분류: 보안 정책 / 다운로드 세션 관리 상태: 확정

    1. 개요¶

    1.1 목적¶

    본 문서는 다운로드 세션 발급 이후 권한 박탈, 공유 링크 폐기, 정책 변경이 발생했을 때의 시스템 동작 방식과 그 설계 근거를 정의한다.

    1.2 적용 범위¶

    대상 설명
    인증 사용자 다운로드 세션 로그인 사용자가 파일 접근 권한으로 발급받은 세션
    공유 링크 기반 다운로드 세션 공유 링크를 통해 발급된 비인증/인증 세션
    정책 변경 영향권 세션 다운로드 허용 정책 변경 시점에 이미 활성 상태인 세션

    1.3 핵심 원칙 요약¶

    다운로드 세션은 발급 시점의 권한을 기준으로 TTL 동안 유효하며,
발급 이후의 권한 변경에 대한 즉시 revoke는 지원하지 않는다.

    2. 즉시 Revoke 미지원 정책¶

    2.1 정책 정의¶

    본 시스템은 다운로드 세션 발급 이후 권한 박탈 또는 공유 링크 폐기 시, 이미 발급된 세션에 대한 즉시 revoke를 지원하지 않는다.

    다음 상황이 발생하더라도 이미 발급된 다운로드 세션은 남은 TTL 동안 유효하다.

    • 인증 사용자의 파일 접근 권한이 세션 발급 직후 박탈된 경우
    • 공유 링크가 세션 발급 직후 비활성화되거나 폐기된 경우
    • 다운로드 허용 정책이 세션 발급 이후 변경된 경우

    2.2 동작 흐름¶

    [세션 발급 시점]          [권한 박탈 시점]          [세션 만료 시점]
      │                        │                        │
      ▼                        ▼                        ▼
 ─────●━━━━━━━━━━━━━━━━━━━━━━━━●━━━━━━━━━━━━━━━━━━━━━━━━●─────
      │          세션 유효 (최대 5분)                     │
      │                        │                        │
      │  권한 박탈 발생해도     │  세션은 남은 TTL 동안   │
      │  즉시 차단하지 않음     │  계속 유효              │
      │                                                 │
      └─── 발급 시점 권한 기준 ──────────────────────────┘

 ─────────────────────────────────────────────────────────────
      세션 만료 후 재요청 시 → 권한 재검증 → 세션 재발급 필요

    2.3 세션 상태 판정 기준¶

    시점 권한 상태 세션 유효 여부 다운로드 가능 여부
    세션 발급 시점 권한 보유 유효 가능
    세션 TTL 내 · 권한 박탈 후 권한 없음 유효 (TTL 잔여) 가능
    세션 TTL 만료 후 권한 없음 만료 불가 (재검증 필요)
    세션 TTL 만료 후 권한 보유 만료 불가 (재발급 필요)

    3. 설계 근거¶

    3.1 세션의 특성¶

    즉시 revoke를 기본 동작으로 두지 않는 이유는 다운로드 세션이 일반적인 API access token이 아니라, 브라우저 다운로드 동작을 수용하기 위한 짧은 수명의 스트리밍 세션이기 때문이다.

    ┌──────────────────────────────────────────────────────────────┐
│                    다운로드 세션 설계 전제                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  • 세션은 짧게 유지하되, 살아 있는 동안은 다음을 허용한다:     │
│    - 브라우저의 동일 URL 재요청                                │
│    - Range 헤더 기반 이어받기 (resume)                         │
│    - 네트워크 오류 후 재시도                                   │
│                                                              │
│  • 다음은 지원하지 않는다:                                     │
│    - 한 HTTP 요청 안의 multi-range                            │
│    - 병렬 Range 다운로드 (다중 커넥션 분할)                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘

    이 전제 아래에서 "발급 직후 즉시 무효화 가능" 정책은 브라우저 호환성과 충돌할 수 있다.

    3.2 보안 모델 선택¶

    본 시스템은 두 가지 보안 모델 중 후자를 채택하였다.

    ┌─────────────────────────────────┐    ┌─────────────────────────────────┐
│   모델 A: 중앙 강제 차단 방식     │    │   모델 B: 단명 세션 방식 (채택)  │
├─────────────────────────────────┤    ├─────────────────────────────────┤
│                                 │    │                                 │
│  • 세션을 길게 유지              │    │  • 세션 자체를 짧게 유지 (≤5분)  │
│  • 중앙에서 실시간 강제 차단     │    │  • 노출 시간을 구조적으로 제한   │
│  • 매 요청마다 세션 유효성 조회  │    │  • TTL 만료 후 자동 재검증       │
│                                 │    │                                 │
│  장점: 즉시 revoke 가능         │    │  장점: 단순, 브라우저 호환       │
│  단점: 조회 부하, 복잡도 증가   │    │  단점: TTL 내 즉시 차단 불가     │
│                                 │    │                                 │
└─────────────────────────────────┘    └─────────────────────────────────┘

    3.3 세션 바인딩에 의한 재생 공격 완화¶

    다운로드 세션 = f(파일 ID, 요청자 식별자, 발급 시각)

  ┌────────────────┐
  │  세션 바인딩    │──→ 파일 ID         : 특정 파일에만 유효
  │  (replay 방지) │──→ 요청자 식별자    : 발급 대상자만 사용 가능
  │                │──→ 짧은 TTL (≤5분) : 시간 제한으로 노출 최소화
  │                │──→ 재요청 허용      : 정상 브라우저 동작 수용
  └────────────────┘

    4. 최대 허용 노출 시간¶

    4.1 TTL 상한¶

    즉시 revoke를 지원하지 않는 대신, 다운로드 세션의 최대 허용 노출 시간은 5분으로 제한한다.

    항목 값
    세션 TTL 상한 300초 (5분)
    TTL 만료 후 동작 세션 자동 무효화
    재다운로드 시 권한 재검증 → 사전 검증 → 세션 재발급

    4.2 세션 만료 후 재발급 흐름¶

    세션 만료 후 동일 요청자가 다시 다운로드를 시도하는 경우:

  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
  │  권한 검증   │────→│  사전 검증   │────→│  세션 발급   │────→│  스트리밍    │
  │             │     │             │     │             │     │             │
  │ 접근 권한   │     │ 파일 상태   │     │ 새 TTL 부여  │     │ Range/GET   │
  │ 재평가     │     │ 다운로드    │     │ 바인딩 생성  │     │ 요청 처리   │
  │             │     │ 가능 여부   │     │             │     │             │
  └─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
        │                    │
        ▼                    ▼
   권한 박탈 상태면       파일 삭제/비활성이면
   여기서 차단됨          여기서 차단됨

    이 시점에서 권한이 이미 박탈되었거나 공유 링크가 폐기된 상태라면 세션 재발급이 거부된다.

    5. TTL 정책 확정¶

    5.1 세션 운영 규칙¶

    규칙 설명
    단명 토큰 운영 세션 TTL은 최대 5분으로 제한한다
    바인딩 필수 세션은 파일 ID와 요청자 식별자에 바인딩한다
    TTL 내 재요청 허용 GET, Range, resume 요청은 동일 세션으로 허용한다
    만료 후 재검증 필수 세션 만료 후에는 반드시 권한·상태 재검증을 거쳐 새 세션을 발급받아야 한다

    5.2 TTL 내 허용·불허 요청¶

    TTL 유효 기간 내:
  ✅ 허용                          ❌ 불허
  ─────────────────────────        ─────────────────────────
  동일 URL GET 재요청               multi-range 요청
  Range 헤더 이어받기               병렬 커넥션 분할 다운로드
  네트워크 재시도                    다른 파일 ID 접근
  부분 콘텐츠 (206) 응답            다른 요청자로의 세션 전용

    6. 보안 사고 대응 시 강제 만료¶

    6.1 현재 정책¶

    현재 보안 사고 대응 절차에는 기존 다운로드 세션을 강제로 만료시키는 별도 단계는 포함하지 않는다.

    다음과 같은 중앙 강제 조치는 기본 기능으로 제공하지 않는다.

    미지원 기능 설명
    사용자 세션 일괄 무효화 특정 사용자에게 발급된 모든 다운로드 세션 즉시 차단
    공유 링크 세션 일괄 차단 특정 공유 링크로 발급된 모든 다운로드 세션 즉시 차단
    파일 세션 전체 강제 만료 특정 파일에 대한 모든 활성 다운로드 세션 즉시 차단

    6.2 보안 사고 발생 시 대응 흐름¶

    보안 사고 발생
      │
      ▼
┌──────────────────────┐
│  운영자 즉시 조치     │
│                      │
│  • 공유 링크 폐기     │
│  • 사용자 권한 회수   │
│  • 다운로드 정책 변경 │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────────────────────────────────┐
│  이미 발급된 활성 세션                              │
│                                                    │
│  → 즉시 차단 불가                                  │
│  → TTL 잔여 시간 동안 유효 (최대 5분)               │
│  → 자연 만료에 의존                                 │
│                                                    │
│  ⚠ 이 구간이 현재 정책의 명확한 한계이다             │
└──────────┬───────────────────────────────────────-─┘
           │
           ▼
┌──────────────────────┐
│  세션 자연 만료 이후   │
│                      │
│  • 재요청 시 권한 재검증 발생  │
│  • 이미 박탈된 권한으로 차단   │
│  • 새 세션 발급 거부           │
└──────────────────────┘

    6.3 한계 인식¶

    ⚠ 현재 정책의 명확한 한계

    보안 사고 발생 시에도 이미 발급된 다운로드 세션은 TTL 종료 전까지 유효할 수 있다. 운영자는 링크 폐기나 권한 회수만 수행하고, 세션 자체는 자연 만료에 의존하게 된다.

    세션 메타데이터를 Redis/DB에 선택적으로 저장할 수 있으나, 강제 revoke 절차는 현재 정의되어 있지 않다.

│                     다운로드 세션 Revoke 정책 요약                     │

    부록 A. 정책 결정 매트릭스¶

    결정 항목 선택지 채택 여부 근거
    즉시 revoke 지원 매 요청마다 세션 유효성 중앙 조회 미채택 조회 부하, 브라우저 호환성 충돌
    단명 세션 + 자연 만료 TTL ≤ 5분, 만료 후 재검증 채택 단순성, 브라우저 호환, 노출 시간 제한
    보안 사고 시 강제 만료 별도 revoke 엔드포인트 / 배치 미채택 현재 미정의, 향후 검토 가능
    세션 바인딩 파일 ID + 요청자 식별자 채택 재생 공격 완화

    부록 B. 용어 정의¶

    용어 정의
    다운로드 세션 파일 다운로드를 위해 발급되는 단명 스트리밍 토큰
    TTL (Time To Live) 세션 발급 시점부터 자동 만료까지의 유효 시간
    Revoke 만료 전 세션을 강제로 무효화하는 행위
    세션 바인딩 세션을 특정 파일 ID 및 요청자 식별자에 종속시키는 것
    사전 검증 세션 발급 전 파일 상태·다운로드 가능 여부를 확인하는 단계
    Range 요청 HTTP Range 헤더를 사용한 부분 콘텐츠 요청 (이어받기 등)
