# 클린 아키텍처 기준 개발 순서 가이드

> **문서 버전**: 1.0
> **최종 수정일**: 2025-01-13
> **대상 독자**: 백엔드 개발자, 아키텍트, 테크 리드

---
## 1. 목적

클린 아키텍처에서 개발 순서는 **바깥(API, DB, 외부 서비스)부터 만드는 것이 아니라, 안쪽(도메인, 유스케이스)부터 고정하는 것**이 핵심이다.

개발의 기준 원칙은 다음 한 문장으로 요약된다.

> **비즈니스 규칙을 먼저 만들고, 그 규칙이 필요로 하는 기술 요소를 나중에 연결한다.**

### 이 방식의 장점

| 장점 | 설명 |
|------|------|
| **비즈니스 로직 보호** | DB, 프레임워크, 외부 서비스에 오염되지 않는다 |
| **테스트 용이성** | 인프라 없이 핵심 로직을 검증할 수 있다 |
| **기술 교체 유연성** | PostgreSQL → MySQL, Local → S3, REST → gRPC 교체가 쉽다 |
| **조기 검증** | 핵심 로직을 가장 먼저 검증할 수 있다 |

---

## 2. 기본 원칙

클린 아키텍처에서 **의존성 방향은 항상 안쪽을 향한다.**

```
[API / 프레젠테이션]
        ↓
[인프라스트럭처] ──→ [애플리케이션 (유스케이스 + 포트)]
                              ↓
                       [도메인 (엔티티 + 규칙)]
```

### 각 계층의 역할

| 계층 | 핵심 질문 | 책임 |
|------|-----------|------|
| **코어 (도메인 + 애플리케이션)** | "무엇이 필요한가" | 비즈니스 규칙 정의, 유스케이스 흐름, 포트(인터페이스) 선언 |
| **인프라** | "그것을 어떻게 구현할까" | 포트 구현, 데이터 저장, 외부 시스템 연결 |
| **API** | "외부 요청을 어떻게 코어에 전달할까" | HTTP 진입점, DTO 변환, 유스케이스 호출 |

### 절대 규칙

- **도메인/유스케이스는 인프라를 몰라야 한다**
- **인프라는 코어가 정의한 인터페이스를 구현해야 한다**
- **API는 코어 유스케이스를 호출하는 얇은 진입점이어야 한다**

## 3. 권장 개발 순서

### 3.1 요구사항과 유스케이스 정리

가장 먼저 해야 할 일은 기술 선택이 아니라 **유스케이스 정의**다.

#### 유스케이스 예시

- 회원 가입
- 프로젝트 생성
- 파일 업로드 시작
- 공유 링크 발급
- 다운로드 세션 생성

#### 이 단계에서 정리해야 할 항목

| 항목 | 설명 |
|------|------|
| 입력값 | 유스케이스에 필요한 데이터 |
| 출력값 | 유스케이스 실행 결과 |
| 성공 조건 | 정상 처리 기준 |
| 실패 조건 | 예외 및 거부 기준 |
| 핵심 비즈니스 규칙 | 도메인이 보장해야 하는 불변 조건 |
| 트랜잭션 경계 | 트랜잭션이 필요한지 여부 |

> **이 단계의 목표**: "무엇을 만들 것인가"를 기술이 아니라 **행위 기준**으로 정리하는 것

---

### 3.2 코어 설계

코어는 두 부분으로 나뉜다.

#### 도메인

| 구성 요소 | 예시 |
|-----------|------|
| 엔티티 | `Project`, `User`, `FileItem`, `UploadSession` |
| 값 객체 | `Email`, `ProjectName`, `FileSize` |
| 도메인 규칙 | 상태 전이 조건, 불변 조건 |
| 도메인 서비스 | 여러 엔티티에 걸친 비즈니스 규칙 |

#### 애플리케이션

| 구성 요소 | 예시 |
|-----------|------|
| 유스케이스 | `CreateProjectUseCase`, `StartUploadUseCase` |
| 입력/출력 모델 | `CreateProjectCommand`, `ProjectResult` |
| 포트(인터페이스) | `IProjectRepository`, `IFileStorage`, `IClock` |

#### 포트(인터페이스) 예시

유스케이스가 필요로 하는 외부 기능을 인터페이스로 선언한다.

```
IProjectRepository     — 프로젝트 저장/조회
IUserRepository        — 사용자 저장/조회
IFileStorage           — 파일 저장/조회
IClock                 — 현재 시간 조회
IIdGenerator           — 고유 ID 생성
ITransactionManager    — 트랜잭션 관리
```

> ⚠️ **이 단계에서 중요한 점**: 아직 DB 구현, Redis 구현, S3 구현을 하지 않는다. 먼저 코어가 "무엇을 필요로 하는지"만 정의한다.

---

### 3.3 코어 테스트 작성

코어를 만든 뒤 **바로** 테스트한다.

#### 테스트 대상

- 도메인 규칙 테스트
- 유스케이스 테스트
- 예외/실패 케이스 테스트

#### 테스트 방식

| 방식               | 설명                                     |
| ---------------- | -------------------------------------- |
| Fake/InMemory 구현 | Repository는 Mock보다 Fake가 더 실용적인 경우가 많음 |
| 순수 검증            | 외부 기술 없이 검증                            |
| 빠르고 반복 가능        | 밀리초 단위 실행                              |

#### 테스트 예시

- 정상 프로젝트 생성
- 이미 같은 이름의 프로젝트가 있으면 생성 실패
- 권한이 없으면 공유 링크 발급 실패
- Quota 초과면 업로드 시작 실패

> **이 단계의 목표**: 핵심 비즈니스 로직이 **인프라 없이도 성립하는지** 검증하는 것
>
> 클린 아키텍처에서 가장 먼저 신뢰해야 하는 것은 API도 아니고 DB도 아니라 **코어 테스트**다.

---

### 3.4 인프라 구현

코어 테스트가 통과하면 그다음에 인프라를 붙인다.

코어에서 정의한 포트를 실제 기술로 구현한다.

#### 구현 매핑 예시

| 포트 (코어 인터페이스) | 인프라 구현체 |
|----------------------|---------------|
| `IProjectRepository` | `PostgresProjectRepository` |
| `IFileStorage` | `LocalFileStorage` / `S3FileStorage` |
| `IClock` | `SystemClock` |
| `IEventPublisher` | `RedisEventPublisher` |
| `IIdGenerator` | `UuidGenerator` |

#### 인프라 구현 원칙

| 원칙 | 설명 |
|------|------|
| 코어 인터페이스를 구현한다 | 인프라가 코어를 의존 |
| 코어 규칙을 바꾸면 안 된다 | 비즈니스 판단은 인프라에 넣지 않음 |
| 저장/전달/연결 책임만 가진다 | 데이터 저장, 외부 호출, 메시지 발행, 파일 저장 등 |

> ⚠️ **비즈니스 판단은 인프라에 넣지 않는다.**

---

### 3.5 API / 프레젠테이션 구현

#### API 계층에 포함되는 것

- Controller / Endpoint
- Request DTO
- Response DTO
- 인증/인가 연결
- 예외 → HTTP 응답 변환
- 유스케이스 호출

#### API의 책임 범위

| ✅ API가 해야 하는 것 | ❌ API에서 하면 안 되는 것 |
|----------------------|-------------------------|
| 입력 검증 | 핵심 비즈니스 규칙 작성 |
| DTO → Command 변환 | 복잡한 if/else 분기 처리 |
| 유스케이스 실행 | 트랜잭션 판단 |
| 결과 → Response 변환 | 저장소 직접 조작 |
| 예외 → HTTP 상태 코드 매핑 | 도메인 로직 직접 수행 |

> **API는 시스템 입구일 뿐이고, 판단은 코어가 해야 한다.**

---

### 3.6 통합 테스트

마지막으로 실제 연결 상태를 검증한다.

#### 검증 대상

- API + UseCase + Repository 전체 흐름
- DB 연결 및 매핑
- 외부 서비스 연동
- 파일 저장
- 트랜잭션 처리
- 직렬화/역직렬화
- 인증/인가 흐름

#### 확인 사항

- 요청부터 응답까지 전체 흐름이 정상 동작하는가
- DB 매핑이 올바른가
- 인프라 구현이 코어 계약을 제대로 만족하는가
- HTTP 상태 코드와 오류 응답 형식이 일관적인가

## 4. 전체 순서 요약

```
┌─────────────────────────────────────────────────────────────┐
│                      개발 순서                               │
│                                                             │
│  ① 유스케이스 정의                                           │
│       ↓                                                     │
│  ② 도메인 / 애플리케이션 코어 설계                             │
│       ↓                                                     │
│  ③ 포트(인터페이스) 정의                                      │
│       ↓                                                     │
│  ④ 코어 테스트 작성 및 통과                                    │
│       ↓                                                     │
│  ⑤ 인프라 구현                                               │
│       ↓                                                     │
│  ⑥ API 구현                                                 │
│       ↓                                                     │
│  ⑦ 통합 테스트                                               │
│       ↓                                                     │
│  ⑧ 운영 관점 보완                                            │
│     • 로깅                                                   │
│     • 모니터링                                               │
│     • 성능                                                   │
│     • 보안                                                   │
│     • 배포 설정                                              │
└─────────────────────────────────────────────────────────────┘
```

| 단계 | 작업 | 핵심 산출물 |
|------|------|------------|
| ① | 유스케이스 정의 | 유스케이스 목록, 입출력 정의, 규칙 정리 |
| ② | 코어 설계 | 엔티티, 값 객체, 유스케이스, 도메인 서비스 |
| ③ | 포트 정의 | 인터페이스 (Repository, Storage, Clock 등) |
| ④ | 코어 테스트 | 단위 테스트, Fake 구현 |
| ⑤ | 인프라 구현 | 포트 구현체 (DB, 파일, 메시지 등) |
| ⑥ | API 구현 | Controller, DTO, 예외 매핑 |
| ⑦ | 통합 테스트 | E2E 테스트, DB 포함 테스트 |
| ⑧ | 운영 보완 | 로깅, 모니터링, 보안, 배포 |

## 5. 실전 권장 방식 — 세로 슬라이스

실무에서는 전체 코어를 다 만든 뒤 인프라를 붙이는 방식보다 **유스케이스 단위로 세로 슬라이스(Vertical Slice)** 로 가는 것이 더 좋다.

### 세로 슬라이스 방식

```
기능 A                    기능 B                    기능 C
┌──────────┐             ┌──────────┐             ┌──────────┐
│ API      │             │ API      │             │ API      │
│ UseCase  │             │ UseCase  │             │ UseCase  │
│ Domain   │  ──완성──→   │ Domain   │  ──완성──→   │ Domain   │
│ Infra    │             │ Infra    │             │ Infra    │
│ Test     │             │ Test     │             │ Test     │
└──────────┘             └──────────┘             └──────────┘
```

하나의 기능(예: `프로젝트 생성`)을 기준으로:

1. 유스케이스 정의
2. 필요한 도메인/포트 작성
3. 코어 테스트 작성
4. 저장소 구현
5. API 연결
6. 통합 테스트

그다음에 `프로젝트 초대`, `프로젝트 참여`, `회의 생성`처럼 확장한다.

### 세로 슬라이스의 장점

| 장점 | 설명 |
|------|------|
| **빠른 피드백** | 기능 하나가 빨리 동작한다 |
| **구조 검증** | 아키텍처가 실제로 맞는지 조기에 확인 가능 |
| **과설계 방지** | 필요한 만큼만 만든다 |
| **MVP 유리** | 최소 기능 단위로 배포 가능 |

> 문서상 순서는 **계층 기준**으로 이해하되, 실제 개발은 **기능 하나를 끝까지 완성하는 방식**이 가장 효율적이다.

## 6. 예시 흐름 — 프로젝트 생성

### 6.1 유스케이스 정의

| 항목 | 내용 |
|------|------|
| **입력** | 프로젝트 이름, GitHub URL |
| **성공** | 프로젝트 생성 완료, 프로젝트 ID 반환 |
| **실패** | 이름 중복, 권한 없음, 입력값 오류 |

### 6.2 코어 작성

```
도메인:
  - Project 엔티티

유스케이스:
  - CreateProjectUseCase

포트:
  - IProjectRepository
  - IIdGenerator
```

### 6.3 코어 테스트

| 테스트 케이스 | 기대 결과 |
|--------------|-----------|
| 정상 생성 | ✅ 프로젝트 생성 성공, ID 반환 |
| 이름 중복 | ❌ `DuplicateProjectNameException` |
| 잘못된 입력 | ❌ `ValidationException` |

### 6.4 인프라 구현

```
IProjectRepository  →  PostgresProjectRepository
IIdGenerator        →  UuidGenerator
```

### 6.5 API 구현

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `POST /projects` |
| **요청** | `{ "name": "...", "githubUrl": "..." }` |
| **성공 응답** | `201 Created` + `{ "id": "...", "name": "..." }` |
| **실패 응답** | `409 Conflict` / `400 Bad Request` |

### 6.6 통합 테스트

- 실제 DB 기준 생성 검증
- 중복 시 409 응답 검증
- 응답 JSON 구조 검증

## 7. 흔한 실수와 안티패턴

### 7.1 DB부터 설계하고 코어를 끌려가게 만드는 것

> 엔티티가 테이블 구조 복사본이 되기 쉽다.
> 클린 아키텍처에서는 **비즈니스 모델이 먼저**다.

### 7.2 Controller에 비즈니스 로직을 넣는 것

> API는 얇아야 한다.
> 조건 분기와 핵심 판단이 Controller에 쌓이면 클린 아키텍처가 무너진다.

### 7.3 인프라 구현을 기준으로 인터페이스를 만드는 것

> ❌ `S3FileStorageService`가 있으니 거기에 맞춰 코어 인터페이스 정의
>
> ✅ **코어가 원하는 행위 기준으로 인터페이스를 정의**해야 한다.

### 7.4 도메인만 만들고 유스케이스를 늦게 만드는 것

> 엔티티만 먼저 과하게 만들면 실사용 흐름과 분리될 수 있다.
> 도메인과 유스케이스는 **함께 발전**하는 게 좋다.

### 7.5 모든 것을 한 번에 설계하려는 것

> 처음부터 전체 시스템 완벽 설계보다
> **핵심 유스케이스 하나를 끝까지 관통**하는 편이 훨씬 낫다.

---

## 8. 최종 가이드 문장

### 원칙

> **유스케이스를 정의하고, 이를 수행하는 코어(도메인 · 애플리케이션 · 포트)를 먼저 설계하고 테스트한 뒤, 그 포트를 구현하는 인프라를 붙이고, 마지막에 API를 연결한다.**

### 실무 적용

> **유스케이스 하나를 기준으로 코어 → 테스트 → 인프라 → API → 통합 테스트 순으로 완성해 나가는 방식이 가장 현실적인 클린 아키텍처 개발 순서다.**
