
본 시스템은 업로드 시작 전, 대상 폴더 기준으로 기존 `FileItem`과 활성 상태의 `FileReservation`을 함께 검사하여 파일명 충돌을 사전에 차단한다. 그러나 사전 검사를 통과했더라도 finalize 직전 또는 최종 반영 단계에서 파일명 충돌이 다시 발생할 수 있으므로, 이에 대한 최종 정책을 명확히 정의해야 한다.

### 사전 검사 이후에도 충돌이 발생할 수 있는 시나리오

사전 검사에서 충돌이 없더라도 다음과 같은 상황에서는 최종 충돌이 발생할 수 있다.

- 동일 폴더에 대해 거의 동시에 두 개 이상의 업로드가 시작된 경우
    
- 사전 검사 직후 다른 세션이 먼저 finalize를 완료한 경우
    
- 예약 상태 정리 지연 또는 비정상 세션 복구 과정에서 동일 이름이 뒤늦게 확정된 경우
    
- 운영자 개입, 데이터 보정, 외부 동기화 등으로 업로드 진행 중 동일 이름 파일이 생성된 경우
    

즉, 사전 검사는 충돌 가능성을 줄이기 위한 1차 방어선이지만, **최종 저장 확정 시점의 이름 유일성까지 완전히 보장하는 수단은 아니다.** 따라서 최종 반영 단계에서도 동일 폴더 내 파일명 충돌 여부를 다시 확인해야 한다.

### 최종 충돌 처리 정책

본 시스템은 최종 파일명 충돌 발생 시 **업로드를 실패로 반환하는 정책**을 채택한다.

즉, 아래 정책은 채택하지 않는다.

- 자동 리네임 `(1)`, `(2)` 부여
    
- 버전 업로드로 자동 전환
    
- 동일 이름 파일 덮어쓰기
    
- 서버 측 임의 이름 변경
    

최종 충돌이 발생하면 서버는 해당 업로드를 확정하지 않고 실패 처리하며, 클라이언트는 사용자에게 동일 이름 파일이 이미 존재함을 안내받는다.

이 정책을 채택하는 이유는 다음과 같다.

첫째, 파일명은 사용자 관점에서 단순 식별자가 아니라 저장 위치와 맥락을 포함한 명시적 선택 결과이므로, 서버가 임의로 이름을 바꾸면 사용자가 의도하지 않은 파일 구조가 생성될 수 있다.

둘째, 자동 리네임은 구현은 단순해 보일 수 있으나, 실제로는 동기화·공유·링크·후속 조회에서 사용자가 어떤 파일이 원본인지 혼동할 가능성이 크다.

셋째, 덮어쓰기나 버전 업로드는 별도의 명시적 사용자 의도와 UI 설계가 필요한 기능이므로, 기본 업로드 동작에 암묵적으로 포함시키지 않는다.

따라서 본 시스템의 기본 정책은 **“충돌 시 실패”**이며, 파일명 재선택 책임은 클라이언트와 사용자에게 둔다.

### 최종 충돌 판정 시점

최종 충돌 검사는 최소한 다음 두 시점에서 수행한다.

- 업로드 시작 전 사전 검사 시점
    
- finalize 직전 또는 `FileItem` 생성 직전 최종 확정 시점
    

특히 최종 확정 시점에는 DB 반영 직전에 동일 폴더 내 활성 파일명 유일성이 다시 검증되어야 하며, 가능하다면 DB 유니크 제약 또는 이에 준하는 보호 장치로 중복 생성을 방지해야 한다.

즉, 사전 검사 실패는 사용자 경험 개선을 위한 빠른 피드백이고, 최종 검사 실패는 동시성 경쟁 상황에서의 정합성 보장을 위한 안전장치다.

### 충돌 발생 시 사용자 경험(UX) 시나리오

최종 충돌이 발생하면 클라이언트는 업로드를 성공으로 표시하지 않고 실패로 처리한다.  
사용자에게는 다음과 같은 메시지를 제공한다.

- 기본 메시지: `같은 이름의 파일이 이미 존재하여 업로드할 수 없습니다.`
    
- 안내 메시지: `파일 이름을 변경한 뒤 다시 시도해주세요.`
    

클라이언트 동작은 다음과 같다.

1. 업로드 항목을 실패 상태로 표시한다.
    
2. 실패 원인을 “파일명 충돌”로 명시한다.
    
3. 사용자에게 다른 이름으로 저장할 것을 안내한다.
    
4. 필요 시 파일명 수정 후 재업로드할 수 있는 흐름을 제공한다.
    

즉, 시스템은 서버 측에서 자동으로 이름을 바꾸지 않고, **사용자에게 다른 이름을 선택하도록 권유하는 UX**를 기본으로 한다.

### 에러 코드 및 서버 응답 정책

최종 파일명 충돌 시 서버는 명확한 비즈니스 에러 코드를 반환해야 한다.

권장 예시는 다음과 같다.

- 에러 코드: `FILE_NAME_CONFLICT`
    
- 사용자 메시지: `같은 이름의 파일이 이미 존재합니다. 다른 이름으로 다시 시도해주세요.`
    

이 에러는 quota 초과, MIME 차단, 저장소 이동 실패 등과 구분되는 독립적인 실패 사유로 취급해야 하며, 운영 로그에는 대상 폴더 ID, 충돌 파일명, 세션 ID 등을 함께 기록한다.

### 예약 및 세션 처리 원칙

최종 충돌로 업로드가 실패한 경우, 해당 업로드는 정상 확정되지 않으므로 예약 자원은 유지되지 않아야 한다.

따라서 다음 처리가 함께 수행되어야 한다.

- `UploadSession -> FAILED`
    
- `FileReservation -> FAILED` 또는 정책상 해제 상태로 전환
    
- `User.storage_reserved_bytes` 차감
    
- 임시 파일 정리 대상 등록 또는 즉시 삭제 시도
    

즉, 파일명 충돌은 단순 응답 실패가 아니라 **업로드 실패로 간주되는 종료 시나리오**이며, quota 예약 해제와 임시 파일 정리까지 포함해 처리해야 한다.

### 운영 원칙 요약

파일명 충돌 정책은 다음 원칙을 따른다.

- 사전 검사에서는 `FileItem`과 활성 `FileReservation`을 함께 검사한다.
    
- 사전 검사 통과 후에도 동시성이나 후행 확정으로 인해 최종 충돌이 발생할 수 있다.
    
- 최종 충돌 발생 시 서버는 자동 리네임, 덮어쓰기, 버전 업로드를 수행하지 않는다.
    
- 최종 정책은 **실패 반환**이다.
    
- 클라이언트는 사용자에게 다른 이름으로 다시 시도할 것을 안내한다.
    
- 충돌 실패 시 예약은 해제되고 세션은 실패 처리된다.
