## 1. 문서 목적

이 문서는 Cloud# 프로젝트를 **모노레포**로 구성할 때의 최상위 디렉토리 구조와 백엔드 Clean Architecture 적용 기준을 정의한다.

### MVP 기술 스택

| 영역 | 선택 |
|------|------|
| 웹 프론트엔드 | React + TypeScript |
| 백엔드 | ASP.NET 기반 모듈형 모놀리스 |
| 백엔드 아키텍처 | Simplified Clean Architecture (`Api / Core / Infrastructure`) |
| 업로드 전송 | tusd |
| 메타데이터 저장소 | PostgreSQL |
| 캐시 / 이벤트 | Redis |
| 파일 저장소 | Local FS |
| 후처리 | `Api` hosted service 기반 media/AI 후처리 |
| 배포 | Docker Compose |

> **핵심 판단**: 초기 MVP에서는 서비스를 과도하게 분리하지 않고, **하나의 백엔드 솔루션 안에서 도메인 경계를 명확히 나누는 방식**을 우선한다.

## 2. 최상위 모노레포 구조

```
cloudsharp/
│
├── apps/
│   ├── web/                    # React 웹 클라이언트
│   ├── mcp-console/            # MCP/AI 연동 콘솔
│   ├── backend/                # ASP.NET 백엔드 솔루션
│   └── upload-gateway/         # tusd 설정 및 hook
│
├── packages/
│   ├── contracts/              # OpenAPI schema, 이벤트 계약
│   ├── api-client/             # Generated API client
│   ├── ui/                     # 공통 UI 컴포넌트
│   └── config/                 # ESLint, Prettier, TS 설정
│
├── infra/
│   ├── compose/                # Docker Compose override
│   ├── docker/                 # Dockerfile
│   ├── postgres/               # DB 초기화, seed, migration
│   ├── redis/                  # Redis 설정
│   ├── nginx/                  # 라우팅 설정
│   └── storage/                # Local FS mount 위치
│
├── docs/
├── tools/
├── .github/
├── .editorconfig
├── .gitignore
├── package.json
├── pnpm-workspace.yaml
├── global.json
├── docker-compose.yml
└── README.md
```

### 2.1 `apps/` — 실행 가능한 애플리케이션

| 경로 | 역할 |
|------|------|
| `apps/web` | 일반 사용자용 React 웹 클라이언트 |
| `apps/mcp-console` | 명령형 콘솔, MCP/AI 연동 진입점 |
| `apps/backend` | ASP.NET 기반 `Api / Core / Infrastructure` 백엔드 솔루션 |
| `apps/upload-gateway` | tusd 설정, hook, 업로드 전송 계층 |

### 2.2 `packages/` — 공유 코드

| 경로 | 역할 |
|------|------|
| `packages/contracts` | OpenAPI schema, 이벤트 계약, 공유 타입 |
| `packages/api-client` | 프론트엔드에서 사용하는 generated API client |
| `packages/ui` | 공통 UI 컴포넌트 |
| `packages/config` | ESLint, Prettier, TypeScript 공통 설정 |

### 2.3 `infra/` — 인프라 파일

| 경로 | 역할 |
|------|------|
| `infra/compose` | Docker Compose override 또는 환경별 compose 파일 |
| `infra/docker` | API, Web용 Dockerfile |
| `infra/postgres` | PostgreSQL 초기화, seed, migration 보조 파일 |
| `infra/redis` | Redis 설정 |
| `infra/nginx` | Web/API/upload/download routing |
| `infra/storage` | Local FS 저장소 mount 위치 |

## 3. 프론트엔드 구조

```
apps/web/
└── src/
    ├── app/                    # 라우터, 전역 Provider, 앱 초기화
    ├── pages/                  # 라우트 단위 화면
    ├── features/
    │   ├── auth/               # 로그인/회원가입
    │   ├── spaces/             # Space 관리
    │   ├── files/              # 파일 탐색
    │   ├── uploads/            # 업로드 기능
    │   └── shares/             # 공유 기능
    ├── entities/
    │   ├── user/
    │   ├── space/
    │   ├── folder/
    │   ├── file/
    │   └── share-link/
    └── shared/
        ├── api/                # API client
        ├── ui/                 # 공통 UI
        ├── hooks/              # 공통 hook
        └── utils/              # 유틸리티
```

### 3.1 구조 기준

| 경로 | 책임 |
|------|------|
| `app` | 라우터, 전역 Provider, 앱 초기화 |
| `pages` | 라우트 단위 화면 |
| `features` | 로그인, Space 관리, 파일 탐색, 업로드, 공유 기능 |
| `entities` | User, Space, Folder, File, ShareLink 모델 |
| `shared` | API client, 공통 UI, hook, 유틸리티 |

> **UX 원칙**: 프론트엔드는 **Space 중심 UX**를 기준으로 구성한다. 사용자는 먼저 서버 또는 Space를 선택하고, 선택된 Space 안에서 파일 탐색, 업로드, 공유, 멤버 관리를 수행한다.

## 4. 백엔드 Simplified Clean Architecture 구조

```
apps/backend/
├── CloudSharp.Backend.sln
├── src/
│   ├── CloudSharp.Api/
│   ├── CloudSharp.Core/
│   └── CloudSharp.Infrastructure/
└── tests/
    ├── CloudSharp.Core.Tests/
    ├── CloudSharp.Infrastructure.Tests/
    ├── CloudSharp.Api.IntegrationTests/
    └── CloudSharp.Architecture.Tests/
```

### 4.1 프로젝트 역할

| 프로젝트 | 역할 |
|----------|------|
| `CloudSharp.Api` | HTTP API, 인증 필터, 요청/응답 매핑, OpenAPI, 백그라운드 작업 host |
| `CloudSharp.Core` | 도메인 모델, 값 객체, 유스케이스, DTO, 포트 인터페이스, 정책 |
| `CloudSharp.Infrastructure` | PostgreSQL, Redis, Local FS, JWT, tusd hook adapter, ffmpeg/AI adapter 구현 |

### 4.2 의존성 방향

```
CloudSharp.Api ──────────→ CloudSharp.Core
                                ↑
CloudSharp.Infrastructure ──────┘
```

**`Api`와 `Infrastructure`는 모두 `Core`에만 의존한다. `Core`는 아무것도 참조하지 않는다.**

#### 금지되는 의존성

| 금지 규칙 | 이유 |
|-----------|------|
| `Core → Api` | 핵심 규칙이 HTTP 표현 계층을 알면 안 된다 |
| `Core → Infrastructure` | 핵심 규칙이 DB, Redis, 파일시스템, 외부 도구를 알면 안 된다 |
| `Api → Infrastructure` 직접 호출 남용 | API는 요청/응답 조립과 DI 구성에 집중하고, 비즈니스 흐름은 Core 유스케이스를 통해 실행한다 |

### 4.3 단순화 기준

기존의 `Domain`과 `Application`을 별도 프로젝트로 나누지 않고 **`Core` 하나로 합친다.**

| 기존 구분 | 단순화 후 위치 |
|-----------|---------------|
| Domain entity, value object, domain policy | `CloudSharp.Core/Domain` |
| Application use case, DTO, port | `CloudSharp.Core/UseCases`, `CloudSharp.Core/Abstractions` |
| Infrastructure adapter | `CloudSharp.Infrastructure` |
| HTTP endpoint, auth filter, OpenAPI | `CloudSharp.Api` |

> **근거**: MVP에서 프로젝트 수와 참조 복잡도를 줄이면서도, 핵심 규칙이 인프라에 의존하지 않는 Clean Architecture의 핵심 원칙은 유지한다.

## 5. 백엔드 내부 디렉토리

### 5.1 `CloudSharp.Core`

```
CloudSharp.Core/
├── Common/
├── Domain/
│   ├── Users/                  # User, SystemRole
│   ├── Spaces/                 # Space, SpaceStatus
│   ├── Memberships/            # SpaceMember, SpaceRole, SpaceInvite
│   ├── Folders/                # Folder, FolderPath
│   ├── Files/                  # FileItem, FileStatus, PreviewStatus, ScanStatus
│   ├── Uploads/                # UploadSession, FileReservation, 상태 전이
│   ├── Shares/                 # ShareLink, ShareLinkTarget, 공유 링크 정책
│   ├── Downloads/              # DownloadSession, 다운로드 세션 상태
│   ├── Quotas/                 # Space quota 계산, 예약 용량 정책
│   └── Audit/                  # AuditLog, 주요 이벤트 타입
├── Abstractions/
│   ├── Auth/
│   ├── Persistence/
│   ├── Storage/
│   ├── Messaging/
│   ├── Clock/
│   └── Processing/
└── UseCases/
    ├── Auth/
    ├── Spaces/
    ├── Members/
    ├── Folders/
    ├── Files/
    ├── Uploads/
    ├── Downloads/
    ├── ShareLinks/
    ├── Quotas/
    └── Admin/
```

`Core`는 기존 `Domain`과 `Application`의 책임을 합친 레이어다. 도메인 모델과 유스케이스를 같은 프로젝트에 두되, **디렉토리로 책임을 분리**한다.

#### Domain 영역

| 영역 | 책임 |
|------|------|
| `Domain/Users` | User, SystemRole |
| `Domain/Spaces` | Space, SpaceStatus |
| `Domain/Memberships` | SpaceMember, SpaceRole, SpaceInvite |
| `Domain/Folders` | Folder, FolderPath |
| `Domain/Files` | FileItem, FileStatus, PreviewStatus, ScanStatus |
| `Domain/Uploads` | UploadSession, FileReservation, 업로드 상태 전이 |
| `Domain/Shares` | ShareLink, ShareLinkTarget, 공유 링크 정책 |
| `Domain/Downloads` | DownloadSession, 다운로드 세션 상태 |
| `Domain/Quotas` | Space quota 계산, 예약 용량 정책 |
| `Domain/Audit` | AuditLog, 주요 이벤트 타입 |

> **제약**: `Core`에는 DB 저장 방식, HTTP 응답 형식, Redis, Local FS, ffmpeg 같은 구현 상세를 두지 않는다. Space 권한, quota 판정, 업로드/예약 상태 전이, 공유 링크 검증 같은 **핵심 규칙과 유스케이스만** 둔다.

#### Core 포트 (Abstractions)

| 포트 | 목적 |
|------|------|
| `ICurrentUser` | 현재 인증 사용자 조회 |
| `IClock` | 시간 의존성 분리 |
| `IStorageService` | Local FS, S3 같은 저장소 추상화 |
| `IEventPublisher` | Redis Pub/Sub 또는 향후 Kafka 추상화 |
| `IUploadSessionRepository` | 업로드 세션 조회/저장 |
| `IFileReservationRepository` | 파일명/quota 예약 조회/저장 |
| `IDownloadSessionTokenService` | 단명 다운로드 세션 토큰 발급/검증 |
| `IPreviewJobRunner` | 미리보기/썸네일 작업 실행 추상화 |
| `IAiMetadataExtractor` | 자동 태그, OCR, 요약 같은 AI 후처리 추상화 |

#### UseCases 영역

| 영역 | 대표 유스케이스 |
|------|--------------|
| `Auth` | 회원가입, 로그인, 프로필 조회 |
| `Spaces` | Space 생성, 목록 조회, 상세 조회, 이름 변경 |
| `Members` | 초대 생성, 초대 수락, Role 변경, 멤버 제거 |
| `Folders` | 폴더 생성, 목록 조회, 이름 변경, 이동, 삭제 |
| `Files` | 파일 목록, 검색, 이름 변경, 이동, 삭제 |
| `Uploads` | 업로드 초기화, finalize, 취소, 만료 처리 |
| `Downloads` | 다운로드 세션 발급, 스트림 권한 검증 |
| `ShareLinks` | 공유 링크 생성, 검증, 비활성화 |
| `Quotas` | quota 조회, quota 변경, 업로드 가능 여부 검사 |
| `Admin` | 전체 Space/사용량 조회 |

### 5.2 `CloudSharp.Api`

```
CloudSharp.Api/
├── Endpoints/
│   ├── Auth/
│   ├── Spaces/
│   ├── Members/
│   ├── Folders/
│   ├── Files/
│   ├── Uploads/
│   ├── Downloads/
│   ├── ShareLinks/
│   └── Admin/
├── Auth/
├── OpenApi/
├── Middlewares/
└── BackgroundServices/
```

`Api`는 외부 요청을 받아 `Core` 유스케이스를 호출하는 **진입점**이다.

| 영역 | 책임 |
|------|------|
| `Endpoints` | HTTP endpoint, request/response DTO mapping |
| `Auth` | 인증/인가 필터, 현재 사용자 컨텍스트 연결 |
| `OpenApi` | Swagger/OpenAPI 문서화 |
| `Middlewares` | 예외 처리, 요청 로그, correlation id |
| `BackgroundServices` | Redis 이벤트 구독, media/AI 후처리 host |

> **확장 전략**: 초기 MVP에서는 별도 Worker 프로젝트를 만들지 않고 `Api`의 hosted service로 후처리를 시작할 수 있다. 추후 부하가 커지면 같은 `Core`와 `Infrastructure`를 참조하는 별도 worker host로 분리한다.

### 5.3 `CloudSharp.Infrastructure`

```
CloudSharp.Infrastructure/
├── Persistence/
│   ├── DbContext/
│   ├── Entities/
│   ├── Configurations/
│   ├── Migrations/
│   └── Repositories/
├── Storage/
│   └── LocalFileStorage/
├── Uploads/
│   └── Tusd/
├── Messaging/
│   └── Redis/
├── Auth/
│   └── Jwt/
├── Preview/
│   └── Ffmpeg/
└── Ai/
    └── MetadataExtraction/
```

`Infrastructure`는 `Core` 포트의 **구현체**를 제공한다.

| 영역 | 구현 책임 |
|------|----------|
| `Persistence` | EF Core DbContext, entity mapping, repository |
| `Storage/LocalFileStorage` | `storage_key`를 실제 Local FS 경로로 resolve |
| `Uploads/Tusd` | tusd hook payload 해석, finalize API 연결 |
| `Messaging/Redis` | Redis Pub/Sub 이벤트 발행 |
| `Auth/Jwt` | JWT 생성, 검증, 인증 설정 |
| `Preview/Ffmpeg` | 미리보기/썸네일 생성 보조 구현 |
| `Ai/MetadataExtraction` | 자동 태그, OCR, 요약 adapter |

## 6. 계약과 코드 생성

```
packages/contracts/
├── openapi/
│   └── cloudsharp.v1.yaml
└── events/
    ├── upload-completed.schema.json
    └── file-preview-requested.schema.json
```

| 계약 | 용도 |
|------|------|
| OpenAPI | Web, MCP Console, 외부 클라이언트 API 생성 |
| Event schema | API와 후처리 hosted service 사이의 이벤트 페이로드 고정 |

> **초기 전략**: OpenAPI를 API 프로젝트에서 생성하고, 결과물을 `packages/contracts/openapi`에 반영하는 방식으로 시작한다.

## 8. 운영 기준

MVP 운영은 Docker Compose 기준으로 시작한다.

```yaml
# docker-compose.yml 서비스 목록
services:
  api:        # ASP.NET 백엔드
  web:        # React 프론트엔드
  tusd:       # 업로드 게이트웨이
  postgres:   # 메타데이터 저장소
  redis:      # 캐시 / 이벤트
  nginx:      # 리버스 프록시
```

### Self-host 전제

| 항목 | 기준 |
|------|------|
| DB/저장소 | 각 서버는 독립 DB와 독립 저장소를 가진다 |
| UX | 프론트엔드는 공통 UX를 제공한다 |
| 검색 | MVP에서는 전체 서버 통합 검색보다 **서버 전환 UX**를 우선한다 |
| 확장 | 추후 OpenSearch, Redis Streams/Kafka, MinIO/S3로 확장한다 |

## 9. 최종 원칙

Cloud# 모노레포는 다음 원칙을 따른다.
