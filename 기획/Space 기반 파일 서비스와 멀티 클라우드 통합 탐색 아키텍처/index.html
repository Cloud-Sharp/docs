    Space 기반 파일 서비스와 멀티 클라우드 통합 탐색 아키텍처

    1. 문서 목적¶

    기존 파일 서비스는 대체로 두 가지 축으로 구성된다.

    첫째는 개인 파일 공간이다. 사용자는 자신의 드라이브나 저장소를 가지고, 파일은 기본적으로 "내 것"으로 관리된다.

    둘째는 공유 폴더다. 개인 파일 공간이나 NAS, 혹은 특정 스토리지 안의 일부 폴더를 다른 사람과 같이 쓰도록 열어주는 방식이다.

    이 방식은 단순하고 익숙하지만, 협업 규모가 커지고 저장 공간이 여러 서비스에 흩어질수록 한계가 드러난다. Cloud#의 Space는 이를 해결하기 위해, 파일 저장과 협업의 기준 단위를 개인 계정이나 공유 폴더가 아니라 독립된 Space 자체로 재정의한다. 문서에서도 Space는 사용자 개인 저장소가 아니라 Space 중심의 완전 분리형 저장 공간으로 설명되며, 파일 업로드/다운로드/공유의 기준 역시 User가 아니라 Space라고 정의된다.

    이 문서는 다음을 하나의 흐름으로 설명한다.

    • 기존 파일 공간과 공유 폴더가 가진 한계
    • Space가 제공하는 구조적 해법
    • 프론트/백엔드 완전 분리의 필요성
    • 여러 클라우드의 Space를 사용자 관점에서 통합 탐색하는 구조

    2. 기존 파일 공간과 공유 폴더의 한계¶

    기존 파일 서비스의 가장 큰 특징은, 저장의 기준이 개인 혹은 폴더라는 점이다.

    2.1 개인 파일 공간의 한계¶

    개인 파일 공간은 혼자 쓰기엔 편하지만, 협업으로 확장될 때 문제가 생긴다.

    • 팀 자료가 특정 개인 계정에 종속될 수 있다
    • 소유자가 바뀌거나 탈퇴하면 관리가 복잡해진다
    • 협업 공간임에도 실제 기준은 개인 저장소다
    • 프로젝트 단위로 권한, 용량, 정책을 관리하기 어렵다

    즉, 개인 파일 공간은 저장의 출발점으로는 좋지만, 지속적인 협업 공간의 최상위 단위로는 애매하다.

    2.2 공유 폴더의 한계¶

    공유 폴더는 기존 개인 파일 공간의 한계를 보완하려는 방식이지만, 본질적으로는 여전히 누군가의 공간 일부를 남에게 열어주는 구조다.

    이 방식은 다음 한계를 가진다.

    • 소유 구조가 개인 중심이라 협업 공간의 독립성이 약하다
    • 내부 협업과 외부 전달이 모두 "공유"로 섞여 보인다
    • 권한이 파일/폴더마다 누적되면 복잡해진다
    • 팀 단위 Quota, 과금, 정책 관리가 애매하다
    • 서비스가 많아질수록 어디에 뭐가 있는지 찾기 어려워진다

    즉, 공유 폴더는 편한 기능이지만, 독립된 협업 공간을 표현하기엔 구조적으로 부족하다.

    3. Space가 기존 구조의 어떤 문제를 해소하는가¶

    Cloud#의 Space는 기존 개인 파일 공간과 공유 폴더가 가진 애매함을 줄이기 위해 설계된 개념이다. 문서에서도 Space는 하나의 독립된 파일 공간이며, Space와 Space는 서로 완전히 독립되고, 하나의 Space는 자체적인 파일/폴더 구조, 멤버, Role, Quota, 공유 링크 정책을 가진다고 설명한다.

    3.1 개인 소유 구조의 한계를 해소한다¶

    기존 공유 폴더가 "누군가의 폴더를 같이 쓰는 구조"라면, Space는 처음부터 "이 Space의 파일"을 중심으로 동작한다. 즉, 파일은 특정 개인보다 Space 소속이 더 중요해진다. 문서에서도 파일은 User 소유 개념보다 Space 소속 개념이 더 중요하다고 명시한다.

    이 구조는 다음을 가능하게 한다.

    • 팀 자료가 개인에게 종속되지 않음
    • 프로젝트 공간을 독립된 단위로 운영 가능
    • 소유자 변경, 멤버 변경, 권한 위임이 더 자연스러움

    3.2 협업 규칙을 공간 단위로 정리한다¶

    Space는 단순 공유가 아니라 멤버십과 Role을 중심으로 협업 구조를 만든다. 문서에서도 Space Member와 Role이 핵심 개념으로 정의되어 있고, 권한은 파일 하나하나보다 Space Membership + Role로 판단한다고 설명한다.

    즉, 기존 공유 폴더가 "같이 쓰긴 하는데 누가 뭘 할 수 있는지 애매한 상태"가 되기 쉬웠다면, Space는:

    • Owner
    • Admin
    • Member
    • Viewer

    같은 구조로 공간 단위 권한을 더 명확히 표현할 수 있다.

    3.3 외부 공유와 내부 협업을 분리한다¶

    문서에서도 Cloud#는 공유를 두 가지로 나눈다.

    • Share Link : 외부 공개/전달
    • Membership : 내부 협업/지속 접근

    즉, "파일 하나를 외부에 보내는 것"과 "같이 일할 사람을 멤버로 초대하는 것"을 구분한다. 이 분리는 기존 공유 폴더 구조에서 자주 섞이던 흐름을 명확히 정리해준다.

    3.4 Space 단위 Quota와 정책을 제공한다¶

    문서에서도 Quota는 개인 단위가 아니라 Space 단위로만 부여한다고 되어 있다. 이 구조는 다음 장점을 갖는다.

    • 팀/프로젝트 단위 용량 관리 가능
    • 사용자별 사용량 계산의 복잡성 감소
    • Space 단위 과금/플랜 설계 용이
    • 정책의 기준점이 명확해짐

    3.5 개인용과 팀용을 하나의 개념으로 통합한다¶

    Space는 반드시 여러 명이 쓰는 협업방이 아니다. 혼자 쓰면 개인 전용 Space, 사람을 초대하면 팀 Space, 링크를 만들면 외부 공유 Space가 된다.

    즉,

    • 개인 파일 공간
    • 공유 폴더
    • 팀 드라이브

    를 따로따로 이해하지 않아도, 하나의 독립된 Space 개념으로 묶을 수 있다.

    4. Space의 확장된 가치: 여러 서비스의 공간을 통합 관리할 수 있다¶

    Space 개념이 강한 이유는 단순히 공유 폴더를 대체하는 데서 끝나지 않는다. Space는 파일의 최상위 관리 단위를 서비스 내부의 폴더가 아니라 독립된 공간으로 끌어올리기 때문에, 서로 다른 클라우드나 스토리지에 있는 공간도 사용자 관점에서 같은 형식으로 다룰 수 있는 가능성을 만든다.

    예를 들어 사용자는 다음처럼 인식할 수 있다.

    • A Cloud / 개인 문서 Space
    • A Cloud / 디자인 Space
    • B Cloud / 프로젝트 Space
    • 사내 스토리지 / 팀 자료 Space

    즉, 더 이상 "어느 서비스의 어느 폴더였더라?"가 아니라 "어느 Space였더라?"로 기억하게 된다.

    이건 서비스가 늘어날수록 더 큰 가치가 된다.

    5. 프론트엔드와 백엔드를 완전히 분리해야 하는 이유¶

    Cloud#가 Space 기반 서비스로 커질수록, 단순 웹앱이 아니라 다음과 같은 다양한 진입점을 가질 가능성이 크다.

    • 웹 클라이언트
    • 모바일 앱
    • 데스크톱 앱
    • 외부 공유 링크 페이지
    • 추후 WebDAV 같은 외부 프로토콜

    이 상황에서 프론트와 백엔드가 강하게 결합되어 있으면 구조가 빠르게 경직된다. 반면 완전 분리 구조를 쓰면 다음 장점이 생긴다.

    5.1 서버 이전이 쉬워진다¶

    • 웹은 정적 파일만 배포하면 된다
    • API 서버는 별도 이전이 가능하다
    • 업로드 서버만 따로 교체할 수 있다
    • 파일 저장소만 별도 전환할 수 있다

    즉, 이전 단위가 작아진다.

    5.2 성격이 다른 요청을 분리 확장할 수 있다¶

    Space 서비스는 한 종류의 요청만 처리하지 않는다.

    • 로그인/인증
    • Space 생성/관리
    • 파일 탐색
    • 정렬/검색
    • 공유 링크 접근
    • 업로드
    • 다운로드/미리보기
    • 비동기 후처리

    이들은 자원 사용 패턴이 다르므로, 분리 구조가 훨씬 유리하다.

    5.3 멀티 클라이언트 대응이 쉬워진다¶

    Space, SpaceMember, ShareLink, UploadSession, FileItem 같은 개념을 API 중심으로 두면, 웹과 모바일과 데스크톱이 같은 핵심 규칙을 공유할 수 있다. 문서의 주요 엔티티 정의와도 잘 맞는다.

    6. 사용자 입장에서의 통합 관리도 쉬워진다¶

    Space를 공통 단위로 잡으면, 백엔드 운영뿐 아니라 사용자 경험 자체도 통합할 수 있다.

    기존에는 사용자가 서비스마다 따로 들어가야 했다.

    • A 클라우드 로그인
    • A 클라우드의 내 파일/공유 폴더 탐색
    • B 서비스 로그인
    • B 서비스의 프로젝트 폴더 탐색
    • NAS 공유 디렉터리 접속

    반면 Space 중심 구조에서는, 사용자에게 "내가 접근 가능한 모든 Space 목록" 을 하나의 화면으로 보여줄 수 있다.

    예를 들면:

    • 최근 사용한 Space
    • 즐겨찾기한 Space
    • 내가 속한 Space
    • 외부 공유받은 Space
    • 서비스별 Space 필터
    • 역할별 Space 필터

    즉, 서비스 중심 탐색이 아니라 공간 중심 탐색이 된다.

    7. 서버 전환 중심 Space 확장 전략¶

    7.1 목표¶

    초기 MVP의 목표는, 여러 클라우드나 여러 self-host 서버의 Space를 완전히 통합 탐색하는 것이 아니라
    사용자가 여러 서버 사이를 쉽게 이동하면서 각 서버의 Space를 자연스럽게 이용할 수 있게 하는 것 이다.

    즉, 처음부터 모든 서버를 하나의 검색/메타데이터/권한 체계로 묶기보다,
    각 서버는 독립적으로 운영하고 프론트엔드가 이를 하나의 진입 경험 으로 연결하는 방식을 우선한다.

    핵심 원칙은 아래와 같다.

    • 각 서버는 독립적으로 운영한다
    • 프론트엔드는 공통된 UI/UX를 제공한다
    • 사용자는 서버를 쉽게 선택하고 전환할 수 있다
    • Space 탐색은 서버 단위로 수행한다
    • 완전한 통합 검색과 통합 권한 체계는 추후 확장 과제로 둔다

    이 접근은 self-host 환경에서 특히 유리하다.
    중앙 집중형 통합 계층 없이도 시작할 수 있고, 서버 수가 적은 초기 단계에서 구현 복잡도를 크게 낮출 수 있기 때문이다.

    7.2 MVP에서의 사용자 경험¶

    MVP에서 중요한 것은 “모든 Space를 하나의 목록으로 합치는 것” 이 아니라,
    사용자가 지금 어느 서버에 들어와 있는지 명확히 이해하고, 다른 서버로 빠르게 이동할 수 있게 하는 것 이다.

    예를 들면:

    • 상단 바에서 현재 서버 표시
    • 서버 선택 드롭다운 제공
    • 최근 접속 서버 목록 제공
    • 즐겨찾기 서버 제공
    • 서버 전환 시 해당 서버의 Space 목록으로 즉시 이동

    즉, MVP의 핵심은 통합 탐색 이 아니라 통합 진입 경험 이다.

    8. 서버 전환 중심 MVP 구조도¶

    이 구조에서 중요한 점은,
    클라이언트가 여러 서버를 같은 방식으로 접근할 수 있게 만들되,
    서버 내부는 서로 완전히 독립적으로 유지한다는 것이다.

    즉, 서버를 하나의 거대한 시스템으로 묶는 것이 아니라
    같은 제품 경험을 가진 여러 독립 노드 로 보는 것이 MVP에 적합하다.

    9. 서버 전환 중심 동작 방식¶

    9.1 서버 목록 진입¶

    사용자는 앱에 들어오면 먼저 자신이 접근 가능한 서버 목록을 본다.

    예:

    • 개인 서버
    • 팀 서버
    • 회사 내부 서버
    • NAS 연동 서버

    이 목록은 전역 통합 Space 목록이 아니라,
    접속 가능한 서버 목록 이다.

    9.2 서버 선택 후 Space 탐색¶

    사용자가 특정 서버를 선택하면, 그 순간부터 탐색은 해당 서버 내부에서만 이루어진다.

    예:

    • Server A 선택 → Server A의 Space 목록 표시
    • Server B 선택 → Server B의 Space 목록 표시
    • NAS Gateway 선택 → NAS 기반 Space 목록 표시

    즉, MVP에서는 “전체 서버를 한 번에 탐색” 하지 않고
    선택된 서버 안에서만 Space를 탐색 한다.

    9.3 검색 범위도 서버 단위¶

    초기 버전에서는 검색도 서버 단위로 한정하는 것이 현실적이다.

    예:

    • Server A 안에서 “기획안” 검색
    • Server B 안에서 “디자인” 검색

    이렇게 하면 중앙 검색 인덱스나 메타데이터 통합 계층 없이도
    충분히 일관된 사용자 경험을 제공할 수 있다.

    전역 검색은 추후 확장 기능으로 둘 수 있다.

    9.4 서버 이동을 가볍게 만든다¶

    MVP에서 중요한 것은 통합 검색보다도 서버 이동 비용을 낮추는 것 이다.

    이를 위해 다음 요소가 유용하다.

    • 상단 서버 선택 메뉴
    • 최근 접속 서버 바로가기
    • 즐겨찾기 서버 고정
    • 서버별 색상/아이콘 구분
    • 현재 접속 중인 서버명 명확 표시

    즉, 사용자가 “지금 어디에 있는지” 와 “어디로 이동할 수 있는지” 를 쉽게 이해하도록 만드는 것이 핵심이다.

    10. 권장 배포 구조¶

    self-host MVP 기준의 권장 구조는 아래와 같다.

    이 구조의 핵심은 다음과 같다.

    • 프론트는 공통 UI를 유지한다
    • 각 서버는 자기 DB와 스토리지를 가진다
    • 업로드/미리보기/검색도 서버 내부에서 처리한다
    • 중앙 통합 백엔드는 두지 않는다
    • 서버를 추가할 때도 독립 노드를 하나 더 붙이는 방식으로 확장한다

    즉, self-host 환경에 맞게 작고 독립적인 서버를 여러 개 운영하는 구조 다.

    11. 이 구조의 기대 효과¶

    11.1 MVP 관점¶

    • 초기 구현 범위를 크게 줄일 수 있다
    • 중앙 통합 검색/권한 번역/메타 집계 없이 시작 가능하다
    • self-host 환경에 맞게 서버별 독립 운영이 가능하다
    • 장애가 발생해도 특정 서버 단위로 격리하기 쉽다

    11.2 사용자 경험 관점¶

    • 여러 서버를 하나의 제품 경험 안에서 접근할 수 있다
    • 서버 이동이 쉬워진다
    • “어느 서비스에 들어가야 하지?” 보다 “어느 서버로 갈까?” 수준으로 단순화된다
    • 서버별로 같은 UI를 쓰기 때문에 학습 비용이 낮다

    11.3 운영 관점¶

    • 각 서버를 별도로 배포하고 이전할 수 있다
    • 서버마다 다른 저장소 정책을 적용할 수 있다
    • 고객별, 조직별, 용도별로 서버를 분리하기 쉽다
    • 향후 특정 서버에만 별도 기능을 실험적으로 붙이기 쉽다

    12. 결론¶

    초기 self-host MVP에서는,
    여러 클라우드와 스토리지를 하나의 통합 탐색 계층으로 묶는 것보다
    각 서버를 독립적으로 운영하면서 사용자가 쉽게 전환할 수 있게 하는 구조 가 더 현실적이다.

    이 방식은 다음 장점을 가진다.

    • 구현 복잡도가 낮다
    • self-host 환경에 잘 맞는다
    • 프론트/백엔드 완전 분리 구조를 유지할 수 있다
    • 서버 단위 확장과 운영이 쉽다
    • 이후 필요할 때 통합 검색이나 통합 메타데이터 계층으로 확장할 여지를 남긴다

    즉, MVP에서는
    “통합 탐색 허브” 보다 “서버 전환이 쉬운 공통 Space 경험” 이 더 적합하다.

    장기적으로는 여기에 다음 기능을 점진적으로 추가할 수 있다.

    • 서버별 최근 Space 집계
    • 여러 서버 즐겨찾기 모아보기
    • 제한적 통합 검색
    • 선택 서버 묶음 보기
    • 이후의 완전한 멀티 서버 탐색

    하지만 첫 단계에서는,
    탐색 통합보다 서버 전환 단순화가 더 큰 가치 를 만든다.

    12. 결론¶

    Cloud#의 Space는 단순한 공유 폴더가 아니라, 개인 파일 공간과 협업 공간을 모두 포괄하는 독립된 파일 공간 단위다. 문서에서도 Space는 개인 저장소가 아닌, 독립된 공간이자 권한과 Quota, 파일 구조, 공유 정책의 기준 단위로 정의되어 있다.

    이 개념은 단순히 기존 파일 서비스의 불편을 줄이는 데서 끝나지 않는다.

    • 개인 소유 기반 공유 구조의 한계를 줄이고
    • 협업 규칙을 더 명확하게 만들며
    • 내부 협업과 외부 공유를 분리하고
    • 정책과 용량의 기준점을 Space로 통합한다

    여기에 프론트/백엔드 완전 분리 구조를 결합하면,

    • 서버 이전이 쉬워지고
    • 장기적으로는 여러 클라우드와 스토리지의 Space를 하나의 경험으로 묶는 멀티 클라우드 통합 탐색 허브로까지 확장할 수 있다.

    즉, Space는 단순한 저장 단위가 아니라, 미래의 멀티 클라우드 파일 탐색과 협업을 위한 공통 추상화 계층이 될 수 있다.
